1. NAS sync can push a stale DB snapshot (WAL not checkpointed) — potential data loss

app/database.py:14-18 enables WAL mode, so recent commits live in local.db-wal until a checkpoint. But sync_to_nas() (app/sync.py:219) and the startup backup/pull in sync_from_nas() copy only the main .db file with shutil.copyfile. A push right after imports can ship a NAS copy missing the latest transactions; another machine then pulls that stale copy and the periodic loop pushes it back — silently losing data. Bonus: the copy runs synchronously inside the asyncio loop.

In /mnt/dev/faresight (FastAPI + SQLite expense tracker), app/database.py enables SQLite WAL mode via a connect-event PRAGMA. app/sync.py syncs the DB to/from a NAS by shutil.copyfile of LOCAL_DB_PATH only — it never checkpoints the WAL, so commits still sitting in local.db-wal are missing from every pushed copy (sync_to_nas), from the .db.bak backup, and the file may be copied mid-write.

Fix: before every copy of the local DB in app/sync.py (sync_to_nas, the backup step and initial push in sync_from_nas), open a short-lived sqlite3 connection to LOCAL_DB_PATH and run PRAGMA wal_checkpoint(TRUNCATE), or better, use sqlite3.Connection.backup() to produce the file to copy. Put the logic in one helper (e.g. _snapshot_local_db) used by all copy sites. Do not change the pull direction (NAS→local) logic. Also move the blocking sync_to_nas() call in _periodic_sync_loop into asyncio.to_thread so it doesn't block the event loop.

Constraints: tests monkeypatch app.sync.NAS_SHARE_PATH, app.sync.LOCAL_DB_PATH, app.sync._OWN_HOSTNAME, app.sync.SYNC_INTERVAL_MINUTES; keep those module-level globals referenced through the existing helpers. Python 3.14 — use Optional[T], not T | None, in any Pydantic/FastAPI signatures.
